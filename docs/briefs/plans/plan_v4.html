Ujjain Kumbh QR System - Plan v4

Budget-optimized phased plan: 20 days testing, 45 days main event, 25 days safe-side support. Keeps v3 feature set, lowers run cost through phased scaling and strict autoscale controls.

Duration: 20 + 45 + 25 Days
Target Population: 200M
SLA: 99.95% (zero planned downtime)
Peak Ready: 8-10M/day
Cost Target: 24L-34L (infra + ops)

1) v4 Strategy (Cheap but Effective)

v3 Features Preserved
  • Same feature coverage as v3: registration, lost, found, help, complaint, app channel, multilingual, control room, OTP and offline sync.
  • Single active region with warm standby for peak windows and disaster recovery.
  • App Service primary compute, not always-on heavy AKS cluster.
  • Autoscale with hard upper caps to stop cost spikes.
  • Pre-scale only for known peak bathing dates (not full 90-day overprovisioning).
  • Testing 20 days runs in low-load mode; only baseline services stay active.
  • Blob lifecycle policy: hot 7 days, cool next 30 days, archive after closure.
  • Last 25 days run in taper mode as crowd decreases and autoscale scales down aggressively.
  • 16x7 core support + 24x7 only on high-risk windows.
  • No feature cuts; only infra and ops right-sizing.

Operations Model

16x7 + Peak 24x7

24x7 command mode only in major snan windows for cost efficiency.

  • Dual route at edge: primary active region + warm standby failover.
  • Canary 5% to 25% to 100% rollout with automatic rollback on SLO breach.
  • Synthetic probes every minute for registration, OTP, lost/found, help closure.
  • Weekly disaster drill and monthly zone-level field simulation.
  • Cost guardrails: daily burn alert, monthly cap alert, service-level spend limits.
  • Control room dashboard: p95 latency, queue lag, OTP success, API error ratio.
  • Known peak dates: mandatory pre-scale T-48 hours and rollback validation T-12 hours.
  • Critical incidents: auto-page on-call, war-room bridge, and 30-minute postmortem draft.
